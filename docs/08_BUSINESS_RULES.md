## 2. 商品规则

### 2.1 商品上下架规则

#### 2.1.1 商品状态定义

| 状态 | 说明 | 前台可见 | 可购买 |
|------|------|----------|--------|
| 草稿（draft） | 新建商品未提交上架，信息可能不完整 | 否 | 否 |
| 上架（on_sale） | 商品已上架，前台可见可购买 | 是 | 是 |
| 下架（off_sale） | 商品已下架，前台不可见不可购买 | 否 | 否 |

#### 2.1.2 上架条件

商品必须同时满足以下所有条件才能上架：

1. **SKU库存**：至少一个SKU有库存（库存 > 0）
2. **商品信息完整**：
   - 商品名称：已填写，长度1-120字符
   - 商品分类：已选择，至少选择到二级分类
   - 商品主图：至少上传1张主图
   - 商品价格：至少一个SKU设置了售价
3. **合规检查**：商品内容不违反平台规范

上架操作：管理员在后台手动执行上架操作，系统自动校验上述条件，校验通过后状态变更为"上架"。

#### 2.1.3 下架规则

1. **下架触发**：
   - 管理员手动下架
   - 所有SKU库存为0时，系统自动下架（可选配置）
2. **下架影响**：
   - 前台不可见，无法搜索到该商品
   - 已加入购物车的该商品标记为"已下架"状态，显示灰色样式
   - 已生成的订单不受影响，继续正常流转
   - 商品详情页返回"商品已下架"提示
3. **重新上架**：下架商品可重新上架，需重新满足上架条件

#### 2.1.4 删除规则

1. **删除方式**：软删除（逻辑删除），数据库中保留记录，标记 `is_deleted = true`
2. **删除条件**：
   - 商品关联的所有订单均已完成或已取消（不存在未完成订单）
   - 不存在进行中的售后申请
3. **不可删除的情况**：
   - 存在待付款、待发货、已发货状态的订单
   - 存在进行中的售后申请
4. **删除影响**：
   - 后台商品列表中不再显示（除非选择查看已删除商品）
   - 前台彻底不可见
   - 历史订单中的商品信息保留快照，不受影响

### 2.2 SKU库存规则

#### 2.2.1 库存扣减时机

| 时机 | 操作 | 说明 |
|------|------|------|
| 下单时 | 预扣库存 | 创建订单时锁定对应SKU库存，其他用户不可购买已被锁定的库存 |
| 支付成功 | 确认扣减 | 支付成功后将预扣库存转为实际扣减，库存永久减少 |
| 取消订单 | 归还库存 | 用户主动取消订单时，释放预扣库存，恢复可售库存 |
| 支付超时 | 自动归还 | 订单超时未支付自动关闭，释放预扣库存 |

#### 2.2.2 库存预警

1. **预警阈值**：每个SKU可单独设置库存预警阈值，默认值为10
2. **预警触发**：当SKU可用库存（总库存 - 已锁定库存）低于预警阈值时，系统在后台首页显示预警提醒
3. **预警通知**：后台消息通知 + 可选的微信通知（通知管理员）
4. **预警处理**：管理员收到预警后应及时补货或调整预警阈值

#### 2.2.3 库存为0时的处理

1. 前台商品详情页显示"暂时缺货"标签
2. "加入购物车"按钮置灰不可点击
3. "立即购买"按钮置灰不可点击
4. 商品在列表中仍可展示，但标注缺货状态
5. 购物车中已有的该SKU标记为"库存不足"

#### 2.2.4 负库存控制

1. **不允许负库存**：系统层面严格禁止库存为负数
2. **并发控制**：高并发下单时通过数据库行锁或分布式锁保证库存扣减的原子性，防止超卖
3. **库存校验**：下单时校验可用库存 ≥ 购买数量，不满足则拒绝下单并提示"库存不足"

### 2.3 商品价格规则

#### 2.3.1 价格类型定义

| 价格类型 | 说明 | 前台可见 | 用途 |
|----------|------|----------|------|
| 成本价（cost_price） | 商品进货成本 | 否，仅供后台查看 | 利润计算、采购参考 |
| 原价（original_price） | 商品标价/划线价 | 是，作为划线价展示 | 价格对比展示 |
| 售价（sell_price） | 实际销售价格 | 是，作为主价格展示 | 订单计算基准 |
| 活动价（activity_price） | 活动期间的特价 | 是，活动期间替代售价 | 活动期间订单计算 |

#### 2.3.2 价格约束规则

1. **售价不能高于原价**：`sell_price ≤ original_price`，否则无法保存
2. **成本价仅供后台查看**：前台和接口均不返回成本价信息
3. **活动价不能高于售价**：`activity_price ≤ sell_price`，否则活动无法生效
4. **价格修改不影响已创建的订单**：订单创建时快照商品价格，后续商品价格变更不影响已有订单的金额
5. **价格精度**：所有价格精确到分（小数点后两位），单位为人民币（元）
6. **最低价格限制**：售价不能低于0.01元

## 3. 购物车规则

### 3.1 加入购物车

#### 3.1.1 加入条件

用户将商品加入购物车时，必须满足以下条件：

1. **商品必须上架**：商品状态为"上架"（on_sale），下架或草稿状态不可加入
2. **SKU必须有库存**：所选SKU的可用库存 > 0
3. **商品未被删除**：商品未被软删除

不满足条件时，给出相应提示：
- 商品已下架 → "该商品已下架"
- 库存不足 → "库存不足，当前可购买X件"
- 商品已删除 → "该商品已不存在"

#### 3.1.2 数量规则

| 规则 | 限制值 | 说明 |
|------|--------|------|
| 单品数量上限 | 99 | 同一SKU在购物车中最多99件 |
| 购物车总数量上限 | 99 | 购物车中所有SKU的总数量不超过99件 |
| 最低数量 | 1 | 每个SKU至少1件 |

#### 3.1.3 重复添加处理

- 同一SKU再次加入购物车时，累加数量而非新增条目
- 累加后数量不超过单品数量上限（99件）
- 超过上限时提示"单个商品最多购买99件"

#### 3.1.4 已下架商品处理

- 已在购物车中的商品被下架后，该商品条目标记为"已下架"状态
- 已下架商品显示灰色样式，无法勾选结算
- 已下架商品不参与价格计算
- 用户可手动删除已下架商品
- 如果商品重新上架，购物车中的标记自动解除

---

### 3.2 购物车价格计算

#### 3.2.1 实时计算原则

1. **每次打开购物车重新获取最新价格**：不使用缓存价格，确保价格准确性
2. **价格变动提示**：当购物车中商品价格与上次加入时不同，显示"价格已变更"提示，并展示最新价格
3. **库存不足提示**：当购物车中某SKU数量超过当前可用库存，显示"库存不足，当前可购买X件"提示，并自动将数量调整为可用库存数

#### 3.2.2 购物车金额展示

- 商品小计 = SKU单价 × 数量
- 购物车合计 = 所有勾选商品的小计之和
- 优惠信息预估：展示预估可用的优惠券和积分抵扣（仅供参考，以结算页为准）

## 4. 订单规则

### 4.1 订单创建规则

#### 4.1.1 创建前置条件

| 条件 | 校验规则 | 不满足时处理 |
|------|----------|-------------|
| 收货地址 | 必须选择有效的收货地址 | 提示"请选择收货地址" |
| 商品上架 | 所有商品必须为上架状态 | 提示"商品XX已下架"，移除该商品 |
| 商品库存 | 所有SKU可用库存 ≥ 购买数量 | 提示"商品XX库存不足"，调整数量 |
| 优惠券 | 优惠券必须满足使用条件（金额、范围、有效期） | 提示"优惠券不可使用"，取消优惠券选择 |
| 积分抵扣 | 积分抵扣不能超过订单金额的30% | 自动调整抵扣积分数至合法范围 |
| 商品数量 | 单次下单商品数量不超过限制 | 提示"单次下单数量超限" |

#### 4.1.2 订单金额计算公式

```
实付金额 = 商品总金额 - 优惠金额 - 积分抵扣 + 运费
```

各组成部分的详细定义见 [4.5 订单金额计算规则](#45-订单金额计算规则)。

### 4.2 库存扣减规则

#### 4.2.1 扣减流程

```
创建订单 → 预扣库存（锁定）→ 等待支付
                                    ↓
                          支付成功 → 确认扣减（永久扣减）
                          支付超时 → 归还库存（释放锁定）
                          用户取消 → 归还库存（释放锁定）
```

#### 4.2.2 详细说明

| 阶段 | 操作 | 说明 |
|------|------|------|
| 创建订单时 | 预扣库存 | 将对应SKU的可用库存减少，锁定库存增加。预扣库存有30分钟有效期 |
| 支付成功时 | 确认扣减 | 将锁定库存转为已售库存，库存永久减少 |
| 取消订单时 | 归还库存 | 将锁定库存释放回可用库存，其他用户可购买 |
| 支付超时时 | 自动归还 | 订单超时关闭后，系统自动释放锁定库存 |

#### 4.2.3 并发安全

- 使用数据库行级锁保证库存操作的原子性
- 下单时先查询库存并加锁，再执行扣减，防止超卖
- 库存不足时立即回滚，返回错误提示

---

### 4.3 支付超时规则

#### 4.3.1 超时时间

- 订单创建后 **30分钟** 内未完成支付，系统自动关闭订单
- 超时时间可在后台配置，默认30分钟

#### 4.3.2 超时关闭后的处理

| 处理项 | 操作 | 说明 |
|--------|------|------|
| 归还库存 | 释放预扣库存 | 将锁定库存恢复为可用库存 |
| 归还优惠券 | 恢复优惠券可用状态 | 优惠券状态从"已使用"变更为"未使用"，可再次使用 |
| 归还积分 | 恢复积分余额 | 将抵扣使用的积分退回用户积分账户 |
| 订单状态 | 变更为"已取消" | 订单状态流转至cancelled终态 |
| 通知用户 | 发送订单关闭通知 | 通过小程序消息通知用户订单已超时关闭 |

#### 4.3.3 超时检测机制

- 使用延时任务（如Redis延时队列或定时任务轮询）检测超时订单
- 检测频率：每分钟扫描一次即将超时的订单
- 超时判断依据：订单创建时间 + 超时时间 < 当前时间

### 4.4 订单状态流转

#### 4.4.1 状态定义

| 状态码 | 状态名称 | 说明 | 是否终态 |
|--------|----------|------|----------|
| pending_payment | 待付款 | 订单已创建，等待用户支付 | 否 |
| paid | 已付款 | 用户已完成支付 | 否 |
| pending_delivery | 待发货 | 订单已付款，等待商家发货 | 否 |
| delivered | 已发货 | 商家已发货，等待用户确认收货 | 否 |
| completed | 已完成 | 用户已确认收货或自动确认 | 是 |
| cancelled | 已取消 | 用户取消或超时自动取消 | 是 |
| aftersale | 售后中 | 用户申请售后，订单进入售后流程 | 否 |

#### 4.4.2 状态流转图

```
                    ┌──────────────────────────────────────┐
                    │           用户取消/超时自动取消          │
                    ▼                                      │
  [pending_payment] ──────→ [cancelled]                    │
        │                                                  │
        │ 用户完成支付                                       │
        ▼                                                  │
        [paid] ────────────────────────────────────────────┤
        │                                                  │
        │ 支付成功后自动                                     │
        ▼                                                  │
  [pending_delivery] ─────────────────────────────────────┤
        │                                                  │
        │ 后台发货                                          │
        ▼                                                  │
      [delivered] ────────────────────────────────────────┤
        │                                                  │
        │ 用户确认收货/自动确认                               │
        ▼                                                  │
     [completed]（终态）                                    │
                                                           
  [paid / pending_delivery / delivered] ──→ [aftersale]
                                            │
                                            │ 售后完成
                                            ▼
                                        回到原状态或退款
```

#### 4.4.3 状态流转详细说明

| 流转路径 | 触发条件 | 触发动作 | 备注 |
|----------|----------|----------|------|
| pending_payment → paid | 用户完成微信支付 | 确认扣减库存、发送支付成功通知 | 支付回调确认 |
| pending_payment → cancelled | 用户主动取消 | 归还预扣库存、归还优惠券、归还积分 | 用户可在待付款列表取消 |
| pending_payment → cancelled | 支付超时（30分钟） | 归还预扣库存、归还优惠券、归还积分 | 系统自动执行 |
| paid → pending_delivery | 支付成功后自动流转 | 更新订单状态 | 无需额外操作 |
| pending_delivery → delivered | 管理员后台发货 | 填写物流信息、通知用户、启动自动确认收货倒计时 | 需填写物流公司和单号 |
| delivered → completed | 用户确认收货 | 触发积分发放、更新会员成长值 | 用户手动确认 |
| delivered → completed | 发货后15天自动确认 | 触发积分发放、更新会员成长值 | 系统自动执行 |
| paid → aftersale | 用户申请售后（未发货） | 创建售后单 | 仅退款 |
| pending_delivery → aftersale | 用户申请售后（未发货） | 创建售后单 | 仅退款 |
| delivered → aftersale | 用户申请售后（已发货） | 创建售后单 | 退货退款 |
| completed → aftersale | 用户申请售后（已完成） | 创建售后单 | 退货退款，确认收货后7天内 |
| cancelled | 终态 | 无 | 不可再变更 |
| completed | 终态 | 无 | 售后除外 |

### 4.5 订单金额计算规则

#### 4.5.1 计算公式

```
实付金额 = 商品总金额 - 优惠金额 - 积分抵扣 + 运费
```

#### 4.5.2 各项计算明细

**1. 商品总金额**

```
商品总金额 = Σ(SKU单价 × 数量)
```

- SKU单价取下单时的售价快照（非实时价格）
- 每个订单项（order_item）记录SKU单价和数量

**2. 优惠金额**

```
优惠金额 = 优惠券抵扣 + 活动优惠
```

- 优惠券抵扣：根据优惠券类型和规则计算（详见第7章）
- 活动优惠：根据参与活动的优惠规则计算（详见第8章）

**3. 积分抵扣**

```
积分抵扣 = 使用积分数 × 积分抵扣比例
积分抵扣比例 = 100积分 = 1元
```

- 积分抵扣不能超过商品总金额 - 优惠金额的30%
- 积分抵扣后的金额精确到分，向下取整

**4. 运费**

```
运费 = 根据运费模板计算（详见第13章）
```

**5. 实付金额**

```
实付金额 = 商品总金额 - 优惠金额 - 积分抵扣 + 运费
```

#### 4.5.3 金额约束

1. **实付金额不能小于0.01元**：如果计算结果小于0.01元，则实付金额为0.01元
2. **金额精度**：所有金额计算精确到分（小数点后两位），中间计算过程保留4位小数，最终结果四舍五入到分
3. **金额快照**：订单创建时将各项金额写入订单记录，后续不再变更
4. **优惠金额上限**：优惠金额 + 积分抵扣 ≤ 商品总金额，确保实付金额（不含运费）≥ 0

#### 4.5.4 计算示例

假设：
- 商品A：售价50元 × 2件 = 100元
- 商品B：售价30元 × 1件 = 30元
- 商品总金额 = 130元
- 优惠券：满100减20元
- 积分抵扣：使用300积分 = 3元（130 × 30% = 39元 > 3元，合法）
- 运费：0元（满100包邮）

```
实付金额 = 130 - 20 - 3 + 0 = 107.00元
```

## 5. 发货规则

### 5.1 发货条件

订单必须同时满足以下条件才能发货：

1. **订单状态为待发货**（pending_delivery）
2. **订单已支付**（支付状态为已支付）
3. 不存在进行中的售后申请

### 5.2 发货操作

#### 5.2.1 发货流程

1. 管理员在后台待发货订单列表中选择待发货订单
2. 填写物流信息：
   - 物流公司（从预设列表中选择，如顺丰、中通、圆通等）
   - 物流单号（必填，手动输入或扫码录入）
3. 确认发货

#### 5.2.2 批量发货

1. 支持通过Excel导入批量发货信息
2. Excel模板包含字段：订单号、物流公司、物流单号
3. 导入时逐行校验：
   - 订单号是否存在且状态为待发货
   - 物流公司是否在预设列表中
   - 物流单号格式是否合法
4. 校验失败的行跳过并在结果中提示，成功的行正常发货
5. 支持部分发货（拆单发货），每个包裹独立填写物流信息

#### 5.2.3 发货后动作

| 动作 | 说明 |
|------|------|
| 订单状态变更 | 从"待发货"变更为"已发货" |
| 通知用户 | 通过小程序消息通知用户"您的订单已发货"，包含物流公司和单号 |
| 启动自动确认倒计时 | 发货后15天自动确认收货倒计时开始 |
| 记录发货时间 | 记录实际发货时间，用于后续时效计算 |

### 5.3 自动确认收货

#### 5.3.1 自动确认规则

1. **自动确认时间**：发货后15天自动确认收货
2. **自动确认条件**：用户未主动确认收货且未申请售后
3. **提前提醒**：自动确认前3天（即发货后第12天），通过小程序消息提醒用户"您的订单即将自动确认收货"
4. **自动确认后动作**：
   - 订单状态变更为"已完成"
   - 触发积分发放
   - 更新会员成长值
   - 评价入口开放

#### 5.3.2 特殊情况处理

| 情况 | 处理方式 |
|------|----------|
| 用户在自动确认前申请售后 | 暂停自动确认倒计时，等待售后处理完成 |
| 用户在自动确认前主动确认 | 立即完成确认，取消倒计时 |
| 物流异常（如长时间无更新） | 管理员可手动延长自动确认时间 |

## 6. 售后规则

### 6.1 售后申请条件

#### 6.1.1 可申请售后的订单状态

| 订单状态 | 可申请售后 | 售后类型 |
|----------|-----------|----------|
| 已付款（paid） | 是 | 仅退款 |
| 待发货（pending_delivery） | 是 | 仅退款 |
| 已发货（delivered） | 是 | 退货退款 |
| 已完成（completed） | 是 | 退货退款 |

#### 6.1.2 申请限制

1. **每个订单项只能申请一次售后**：同一order_item不可重复发起售后申请
2. **申请时限**：确认收货后7天内可申请售后，超时不可申请
3. **售后数量**：申请售后的数量不能超过该订单项的购买数量
4. **售后金额**：申请退款的金额不能超过该订单项的实付金额

### 6.2 售后类型

#### 6.2.1 仅退款

- **适用场景**：订单已付款但未发货
- **退款金额**：该订单项的实付金额
- **处理流程**：用户申请 → 管理员审核 → 审核通过后直接退款

#### 6.2.2 退货退款

- **适用场景**：订单已发货或已完成
- **退款金额**：该订单项的实付金额
- **处理流程**：用户申请 → 管理员审核 → 用户填写退货物流 → 管理员确认收货 → 退款

### 6.3 售后状态流转

#### 6.3.1 售后状态定义

| 状态码 | 状态名称 | 说明 |
|--------|----------|------|
| pending_review | 待审核 | 用户已提交售后申请，等待管理员审核 |
| approved | 已审核 | 管理员审核通过，等待用户操作或直接退款 |
| rejected | 已拒绝 | 管理员审核拒绝，售后关闭 |
| returned | 已退货 | 用户已填写退货物流信息，等待管理员确认收货 |
| refunded | 已退款 | 退款已完成 |
| closed | 已关闭 | 售后已关闭（用户撤销或超时） |

#### 6.3.2 状态流转图

```
[pending_review] ──审核通过──→ [approved] ──仅退款──→ [refunded]（终态）
       │                          │
       │审核拒绝                   │退货退款
       ▼                          ▼
  [rejected]（终态）         [returned] ──确认收货并退款──→ [refunded]（终态）

  任何非终态 ──用户撤销/超时──→ [closed]（终态）
```

#### 6.3.3 状态流转详细说明

| 流转路径 | 触发条件 | 触发动作 |
|----------|----------|----------|
| pending_review → approved | 管理员审核通过 | 通知用户审核结果；仅退款类型直接进入退款流程 |
| pending_review → rejected | 管理员审核拒绝 | 通知用户拒绝原因，售后关闭 |
| approved → returned | 用户填写退货物流信息（退货退款类型） | 记录退货物流公司和单号，通知管理员 |
| returned → refunded | 管理员确认收到退货并同意退款 | 执行退款操作，通知用户退款成功 |
| approved → refunded | 仅退款类型，审核通过后直接退款 | 执行退款操作，通知用户退款成功 |
| 任何非终态 → closed | 用户主动撤销售后申请 | 售后关闭，通知用户 |
| 任何非终态 → closed | 用户超时未操作（审核通过后7天未填写退货物流） | 售后自动关闭，通知用户 |

### 6.4 退款规则

#### 6.4.1 退款金额约束

1. **退款金额不能大于实付金额**：单次退款金额 ≤ 订单项实付金额
2. **多次退款累计**：同一订单的累计退款金额 ≤ 订单实付金额

#### 6.4.2 退款金额计算

| 售后类型 | 退款金额计算 | 说明 |
|----------|-------------|------|
| 仅退款 | 订单项实付金额（扣除积分抵扣部分） | 积分抵扣部分以积分形式归还，不退现金 |
| 退货退款 | 订单项实付金额 | 包含积分抵扣部分也以积分形式归还 |

**仅退款详细计算**：

```
退款现金 = 订单项实付金额 - 订单项积分抵扣金额
归还积分 = 订单项积分抵扣金额对应的积分数
```

**退货退款详细计算**：

```
退款现金 = 订单项实付金额 - 订单项积分抵扣金额
归还积分 = 订单项积分抵扣金额对应的积分数
```

#### 6.4.3 退款方式

1. **原路退回**：通过微信支付退款接口，将退款金额原路退回到用户的微信支付账户
2. **退款时效**：退款申请提交后1-3个工作日到账（微信支付处理时间）
3. **退款失败处理**：微信退款接口返回失败时，记录失败原因，管理员可手动重试

#### 6.4.4 退款后的关联处理

| 处理项 | 操作 | 说明 |
|--------|------|------|
| 归还积分 | 将积分抵扣使用的积分退回用户积分账户 | 如果订单使用了积分抵扣，退款时归还对应积分 |
| 优惠券 | **不归还** | 优惠券使用后不归还，无论售后结果如何 |
| 库存 | 归还库存 | 退款成功后将对应SKU库存恢复（仅退货退款时，商品需入库） |
| 会员成长值 | 扣减对应成长值 | 退款成功后扣减该订单获得的会员成长值 |

## 7. 优惠券规则

### 7.1 优惠券类型

#### 7.1.1 满减券

- **规则**：订单金额满X元减Y元
- **示例**：满100减20元
- **参数**：最低使用金额X、优惠金额Y
- **约束**：Y < X（优惠金额必须小于最低使用金额）

#### 7.1.2 折扣券

- **规则**：订单金额满X元享Z折，最高优惠Y元
- **示例**：满100元享8折，最高优惠50元
- **参数**：最低使用金额X、折扣率Z（0.1-0.99）、最高优惠金额Y
- **计算**：优惠金额 = min(商品总金额 × (1 - Z), Y)
- **约束**：折扣后的价格不能低于0.01元

#### 7.1.3 无门槛券

- **规则**：直接减Y元，无最低消费限制
- **示例**：直接减5元
- **参数**：优惠金额Y
- **约束**：优惠后实付金额（不含运费）不能低于0.01元

### 7.2 优惠券使用条件

#### 7.2.1 基本使用条件

| 条件 | 说明 |
|------|------|
| 最低使用金额 | 订单商品总金额（不含运费）必须 ≥ 优惠券设定的最低使用金额 |
| 有效期 | 使用时必须在优惠券的有效期内 |
| 适用范围 | 商品必须在优惠券的适用范围内 |
| 使用状态 | 优惠券状态为"未使用" |

#### 7.2.2 适用范围类型

| 范围类型 | 说明 |
|----------|------|
| 全场通用 | 适用于所有商品 |
| 指定分类 | 仅适用于指定商品分类下的商品 |
| 指定商品 | 仅适用于指定的商品 |

当适用范围为指定分类或指定商品时，最低使用金额的计算基础为适用范围内商品的金额之和。

#### 7.2.3 使用限制

1. **每个订单只能使用一张优惠券**：不可叠加使用多张优惠券
2. **优惠券与活动叠加**：根据活动配置决定是否允许叠加（详见第8章）
3. **售后不归还**：优惠券使用后不归还（详见6.4.4节）
4. **支付超时归还**：订单超时关闭时归还优惠券（详见4.3.2节）

### 7.3 优惠券领取规则

#### 7.3.1 领取限制

| 限制类型 | 说明 | 默认值 |
|----------|------|--------|
| 每人限领数量 | 同一用户最多领取该优惠券的次数 | 每人限领1张 |
| 总量限制 | 优惠券的总发放数量 | 根据运营需求设定 |
| 会员等级限制 | 仅指定等级及以上的会员可领取 | 无限制（所有等级可领） |
| 新人专属限制 | 仅新注册用户可领取 | 否 |

#### 7.3.2 领取校验流程

1. 校验优惠券是否还有剩余数量
2. 校验用户是否已达到限领数量
3. 校验用户会员等级是否满足要求
4. 校验是否为新人专属（注册7天内为新用户）
5. 校验通过后，优惠券状态变更为"未使用"，关联到用户账户

### 7.4 优惠券过期规则

#### 7.4.1 有效期类型

| 类型 | 说明 | 示例 |
|------|------|------|
| 固定时间段 | 指定起止时间 | 2026-06-01 00:00:00 至 2026-06-30 23:59:59 |
| 领取后N天有效 | 从领取时刻起计算N天 | 领取后30天内有效 |

#### 7.4.2 过期处理

1. **过期标记**：到达有效期截止时间后，系统自动将优惠券标记为"已过期"
2. **过期检测**：每日凌晨定时任务扫描已过期的优惠券并更新状态
3. **过期后不可使用**：已过期的优惠券不可用于下单
4. **过期通知**：优惠券过期前1天通过小程序消息提醒用户"您的优惠券即将过期"

## 8. 活动叠加规则

### 8.1 叠加原则

#### 8.1.1 叠加规则矩阵

| 组合 | 是否可叠加 | 说明 |
|------|-----------|------|
| 优惠券 + 活动 | 默认可叠加，活动可配置是否允许叠加优惠券 | 活动创建时可设置"允许叠加优惠券"开关 |
| 积分抵扣 + 活动 | 可叠加 | 积分抵扣始终可与活动叠加 |
| 多个活动 | 不可叠加 | 同一商品只参与一个最优活动 |
| 会员折扣 + 活动 | 取更优惠的价格，不叠加 | 会员折扣和活动价取低者 |

#### 8.1.2 "最优活动"判定

当同一商品同时符合多个活动条件时，按以下规则选择：

1. 优先选择对用户最优惠的活动（即价格最低的活动）
2. 如果多个活动优惠力度相同，优先选择结束时间最近的活动（促进尽快下单）
3. 用户不可手动选择参与哪个活动

### 8.2 价格计算优先级

#### 8.2.1 计算步骤

按以下顺序逐步计算最终价格：

```
步骤1：确定商品基础价格
  ├── 如果商品参与活动 → 取活动价
  ├── 如果商品未参与活动但有会员折扣 → 取会员折扣价
  └── 否则 → 取原售价

步骤2：计算优惠券抵扣
  └── 在步骤1结果基础上，根据优惠券规则计算抵扣金额

步骤3：计算积分抵扣
  └── 在步骤2结果基础上，根据积分抵扣规则计算抵扣金额

步骤4：加上运费
  └── 最终实付金额 = 步骤3结果 + 运费
```

#### 8.2.2 计算示例

假设：
- 商品售价：100元
- 活动价：80元（参与满减活动）
- 会员折扣：95折（金卡会员）
- 优惠券：满100减15元
- 积分抵扣：500积分 = 5元
- 运费：0元

**情况一：活动允许叠加优惠券**

```
步骤1：取活动价 = 80元（活动价80 < 会员折扣价95，取活动价）
步骤2：优惠券抵扣 = 15元（80元满足满100减15？不满足，优惠券不可用）
      实际：活动价80元 < 优惠券最低使用金额100元，优惠券不可用
步骤3：积分抵扣 = 5元（80 × 30% = 24元 > 5元，合法）
步骤4：实付金额 = 80 - 0 - 5 + 0 = 75元
```

**情况二：活动不允许叠加优惠券，商品未参与活动**

```
步骤1：取会员折扣价 = 100 × 0.95 = 95元
步骤2：优惠券抵扣 = 15元（95元 < 100元，不满足满100减15，不可用）
步骤3：积分抵扣 = 5元（95 × 30% = 28.5元 > 5元，合法）
步骤4：实付金额 = 95 - 0 - 5 + 0 = 90元
```

**情况三：商品未参与活动，无会员折扣**

```
步骤1：取原售价 = 100元
步骤2：优惠券抵扣 = 15元（100元满足满100减15）
步骤3：积分抵扣 = 5元（(100-15) × 30% = 25.5元 > 5元，合法）
步骤4：实付金额 = 100 - 15 - 5 + 0 = 80元
```

## 9. 积分规则

### 9.1 积分获得规则

#### 9.1.1 获取方式及规则

| 获取方式 | 积分数量 | 频率限制 | 说明 |
|----------|----------|----------|------|
| 下单获得 | 实付金额 × 积分倍率 | 每笔订单一次 | 基础倍率：1元 = 1积分；会员等级可享受加成倍率 |
| 每日签到 | 5积分 | 每日1次 | 连续签到递增：第1天5分，第2天6分...第7天11分，第8天重置为5分 |
| 分享商品/活动 | 2积分/次 | 每日上限10积分 | 每日最多分享5次获得积分 |
| 完善宝宝档案 | 10积分 | 一次性 | 每个宝宝档案仅首次完善时获得 |
| 评价商品 | 5积分/条 | 每个订单项1次 | 仅评价一次，追评不额外获得积分 |

#### 9.1.2 积分发放时机

| 获取方式 | 发放时机 | 说明 |
|----------|----------|------|
| 下单获得 | 订单完成后（确认收货） | 订单取消或全额退款时不发放 |
| 签到获得 | 签到成功后即时发放 | — |
| 分享获得 | 分享成功后即时发放 | — |
| 完善宝宝档案 | 保存档案后即时发放 | — |
| 评价获得 | 提交评价后即时发放 | — |

#### 9.1.3 会员等级积分加成

| 会员等级 | 积分倍率 | 说明 |
|----------|----------|------|
| 普通会员 | 1.0x | 基础倍率 |
| 银卡会员 | 2.0x | 下单积分 × 2.0 |
| 金卡会员 | 3.0x | 下单积分 × 3.0 |
| 黑金会员 | 5.0x | 下单积分 × 5.0 |

### 9.2 积分抵扣规则

#### 9.2.1 抵扣规则

| 规则 | 说明 |
|------|------|
| 抵扣比例 | 100积分 = 1元 |
| 单笔订单最高抵扣 | 订单金额（商品总金额 - 优惠金额）的30% |
| 最低抵扣 | 1积分（即0.01元） |
| 抵扣精度 | 向下取整到分 |
| 积分抵扣部分不开具发票 | 积分抵扣的金额部分不包含在发票金额中 |

#### 9.2.2 抵扣计算示例

假设：
- 商品总金额：200元
- 优惠金额：20元
- 用户可用积分：8000积分

```
可抵扣上限 = (200 - 20) × 30% = 54元 = 5400积分
用户可用积分 8000 > 5400，最多使用5400积分抵扣54元
用户选择使用3000积分抵扣30元
实付金额 = 200 - 20 - 30 + 运费
```

### 9.3 积分过期规则

#### 9.3.1 过期规则

| 规则 | 说明 |
|------|------|
| 积分有效期 | 获得后12个月（365天） |
| 过期执行时间 | 每月1日凌晨2:00执行过期清理 |
| 过期前提醒 | 过期前7天通过小程序消息提醒用户 |
| 过期后不可恢复 | 过期积分永久清除，不可找回 |

#### 9.3.2 过期计算逻辑

1. 每月1日扫描用户积分账户中超过12个月的积分记录
2. 按照先进先出（FIFO）原则，优先消耗最早获得的积分
3. 计算已过期但尚未消耗的积分数，将其从用户积分余额中扣除
4. 记录积分过期流水

#### 9.3.3 积分消耗顺序

- 使用积分时（抵扣/兑换），优先消耗即将过期的积分
- 即按过期时间从早到晚消耗

## 10. 会员等级升级规则

### 10.1 等级定义

| 等级 | 等级名称 | 成长值范围 | 等级标识色 | 积分倍率 |
|------|----------|-----------|-----------|----------|

5. **多个宝宝时取并集**：用户有多个宝宝时，推荐所有宝宝月龄段对应商品的并集

## 13. 运费规则

### 13.1 运费计算

#### 13.1.1 运费模板

每个商品可关联一个运费模板，运费模板包含以下配置：

| 配置项 | 说明 | 示例 |
|--------|------|------|
| 默认运费 | 全国默认运费 | 8元 |
| 满额包邮 | 订单金额满X元包邮 | 满99元包邮 |
| 偏远地区加收 | 指定地区加收运费 | 新疆、西藏加收15元 |
| 包邮地区 | 指定地区免运费 | 江浙沪包邮 |

#### 13.1.2 运费计算逻辑

```
1. 判断是否满足满额包邮条件
   ├── 订单商品总金额 ≥ 满额包邮阈值 → 运费 = 0
   └── 订单商品总金额 < 满额包邮阈值 → 继续计算

2. 判断收货地址是否为偏远地区
   ├── 是偏远地区 → 运费 = 默认运费 + 偏远地区加收
   └── 不是偏远地区 → 运费 = 默认运费

3. 判断收货地址是否为包邮地区
   ├── 是包邮地区 → 运费 = 0（覆盖步骤2的结果）
   └── 不是包邮地区 → 保持步骤2的结果
```

#### 13.1.3 多商品运费计算

- 订单中包含多个商品时，取运费最高的商品的运费模板计算
- 不累加多个商品的运费
- 满额包邮判断基于订单商品总金额

#### 13.1.4 运费展示

- 运费在订单确认页展示
- 用户选择/切换收货地址时，运费实时重新计算
- 运费为0时显示"免运费"
