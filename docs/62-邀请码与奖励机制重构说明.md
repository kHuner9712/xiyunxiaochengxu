# 62-邀请码与奖励机制重构说明

## 改了哪些文件

| 文件 | 操作 | 说明 |
|------|------|------|
| `docs/muying-invite-code-migration.sql` | 新建 | 用户表增加 invite_code 字段+唯一索引，老数据补齐脚本 |
| `shopxo-backend/app/service/InviteService.php` | 重写 | 邀请码从 md5 规则改为数据库字段查询；GrantReward 使用事务 |
| `shopxo-backend/app/service/UserService.php` | 修改 | 新用户注册时自动生成 invite_code；注册钩子异常不再静默吞掉 |
| `shopxo-backend/app/service/OrderService.php` | 修改 | 首单奖励异常不再静默吞掉，改为记录日志 |

## 为什么改

### 邀请码问题
1. **invite_code 由 md5(user_id + 'muying_invite') 生成**：规则固定、可枚举，只要知道 user_id 就能算出邀请码
2. **GetInviterByCode 全表扫描用户**：遍历所有用户逐个计算 md5 对比，O(n) 复杂度，用户量大时性能极差
3. **无唯一约束**：理论上不同用户可能生成相同邀请码（虽然 md5 碰撞概率低）

### 奖励发放问题
1. **GrantReward 非原子操作**：积分增加、积分日志、奖励状态更新分步执行，中间异常会导致数据不一致
2. **重复发放风险**：并发场景下同一 reward 可能被多次发放
3. **异常静默吞掉**：OrderService 和 UserService 中的 try-catch 块为空，奖励失败无法排查

## 具体修改

### 1. 邀请码重构

#### 数据库变更
- `sxo_user` 表新增 `invite_code char(8)` 字段
- 存储过程为老用户逐行生成唯一 8 位字母数字邀请码
- 添加 `uk_invite_code` 唯一索引

#### 代码变更
- **GenerateInviteCode()**：新增方法，生成随机 8 位邀请码，20 次重试确保唯一
- **InviteInfo()**：从数据库读取 invite_code，为空时自动生成并写回
- **Poster()**：同上逻辑
- **GetInviterByCode()**：从全表扫描改为 `WHERE invite_code = $code` 索引查询
- **UserService 注册流程**：新用户注册后自动生成 invite_code

### 2. 奖励发放原子性

#### GrantReward 修改
- 使用 `Db::startTrans()` / `Db::commit()` / `Db::rollback()` 事务
- 查询奖励记录时使用 `->lock(true)` 行锁，防止并发重复发放
- 积分增加 + 积分日志 + 奖励状态更新在同一事务内
- 异常时回滚事务并记录日志

#### 异常日志
- OrderService 首单奖励 catch 块：记录 `邀请首单奖励异常 order_id=xxx user_id=xxx error=xxx`
- UserService 注册邀请钩子 catch 块：记录 `邀请注册钩子异常 user_id=xxx error=xxx`

## 迁移步骤

### 1. 执行 SQL 迁移
```bash
mysql -u root -p shopxo < docs/muying-invite-code-migration.sql
```

**重要**：必须先执行迁移脚本，否则 invite_code 字段不存在会导致 PHP 报错。

### 2. 部署代码
直接部署更新后的 PHP 文件即可。

### 3. 验证迁移结果
```sql
-- 检查字段是否添加
DESCRIBE sxo_user;

-- 检查唯一索引
SHOW INDEX FROM sxo_user WHERE Key_name = 'uk_invite_code';

-- 检查老用户是否已补齐邀请码
SELECT id, invite_code FROM sxo_user WHERE invite_code = '' OR invite_code IS NULL;
```

## 风险点

1. **迁移脚本执行时间**：存储过程逐行更新老用户，用户量大时耗时较长，建议低峰期执行
2. **唯一索引冲突**：存储过程中已做唯一性检查，但极端情况下（如手动执行时中断后重跑）可能出现空字符串冲突。迁移前建议备份数据库
3. **前端兼容**：invite.vue 和 my-invite.vue 中有硬编码的默认邀请码（如 'MY2026'、'YX2026ABC'），这些是 data() 中的默认值，接口返回后会覆盖，不影响功能
4. **GoodsFavor 表**：Activity.php Detail 接口中查询收藏状态使用了 `GoodsFavor` 表的 `type='activity'`，需确认该表支持此类型。如果不支持，is_favored 将始终为 false，不影响核心功能

## 如何验证

1. **邀请码生成**：注册新用户，检查 sxo_user.invite_code 是否自动生成
2. **邀请码查询**：调用邀请信息接口，确认返回的 invite_code 与数据库一致
3. **邀请码唯一性**：`SELECT invite_code, COUNT(*) FROM sxo_user GROUP BY invite_code HAVING COUNT(*) > 1` — 应返回空
4. **老用户补齐**：检查老用户的 invite_code 是否已填充
5. **奖励发放原子性**：触发邀请奖励，检查积分、日志、奖励状态三者一致
6. **异常日志**：模拟奖励发放异常，检查 ThinkPHP 日志中是否有对应记录
7. **首单奖励不中断订单**：奖励失败时，订单支付流程应正常完成
