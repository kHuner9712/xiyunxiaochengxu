# 孕禧小程序 — 上线执行手册 (Go-Live Runbook)

> 本手册按时间顺序编写，覆盖发布前 1 天到发布后 24 小时的完整操作流程。
> 每个时间段标注：做什么、看什么、出问题先回退哪里。
> 执行人按手册逐步操作，不要跳步。

---

| 角色 | 职责 | 联系方式 |
|------|------|---------|
| 发布指挥 | 统筹全流程，决定是否继续/回退 | |

| 产品 | 内容验收、品牌检查 | |

## T-1天 — 发布前 1 天

### 1. 参数确认

| 序号 | 操作 | 负责人 | 完成标志 |
|------|------|-------|---------|
| 1 | 填写 `release-values-template.md` 所有"实际值"列 | 发布指挥 | 所有 ✅ 项已填 |
| 2 | 确认 AppID / AppSecret 可用 | 前端开发 | 微信后台可登录查看 |
| 3 | 确认 SSL 证书未过期 | 后端运维 | `openssl x509 -checkend 86400 -noout -in cert.pem` |
| 4 | 确认数据库连接信息 | 后端运维 | `mysql -u USER -p -h HOST DB_NAME -e "SELECT 1"` |
| 5 | 确认客服电话 | 产品 | 非占位符，可拨通 |
| 6 | 确认 Logo 图片已准备 | 产品 | 300×300 PNG + 220×66 PNG |
| 7 | 确认隐私政策/注册协议 URL | 产品 | HTTPS 可访问 |

### 2. 代码冻结

| 序号 | 操作 | 负责人 | 完成标志 |
|------|------|-------|---------|
| 1 | 确认无未合并的功能分支 | 前端开发 | `git branch --no-merged main` 为空 |
| 2 | 确认无 P0 Bug 未修复 | 测试 | uat-bug-log 中无 P0 待修 |
| 3 | 打 tag `v1.0.0-rc1` | 前端开发 | `git tag v1.0.0-rc1` |

### 3. 预检脚本运行

| 序号 | 操作 | 负责人 | 完成标志 |
|------|------|-------|---------|
| 1 | 本地运行 `check-placeholders.sh` | 前端开发 | 退出码 0 |
| 2 | 服务器运行 `check-server.sh` | 后端运维 | 全部 PASS 或仅 WARN |

### 4. 体验版构建

| 序号 | 操作 | 负责人 | 完成标志 |
|------|------|-------|---------|
| 1 | 填入 AppID 到 manifest.json | 前端开发 | mp-weixin.appid 有值 |
| 2 | HBuilderX 构建体验版 | 前端开发 | 构建成功无报错 |
| 3 | 微信开发者工具上传 | 前端开发 | 版本管理中出现开发版 |
| 4 | 设为体验版 | 前端开发 | 真机扫码可打开 |

### 5. UAT 测试

| 序号 | 操作 | 负责人 | 完成标志 |
|------|------|-------|---------|
| 1 | 按 `uat-final-checklist.md` 执行全量测试 | 测试 | 所有 P0 项通过 |
| 2 | 记录 Bug 到 `uat-bug-log-template.md` | 测试 | Bug 已分级 |
| 3 | P0 Bug 修复 + 回归 | 前端开发 | 修复后重新测试通过 |

> ⚠️ **决策点**：如果 UAT 发现 P0 Bug 且 2 小时内无法修复 → 延期发布，不冒险上线。

## T-2小时 — 发布前 2 小时

### 1. 服务器最终准备

| 序号 | 操作 | 负责人 | 完成标志 |
|------|------|-------|---------|
| 1 | 创建 .env 文件 | 后端运维 | `cp .env.example .env` 并修改 |
| 2 | APP_DEBUG=false | 后端运维 | `grep APP_DEBUG .env` → false |
| 3 | 数据库密码一致性确认 | 后端运维 | .env = database.php = MySQL |
| 4 | 执行 migration SQL | 后端运维 | A+B+C 段全部成功 |
| 5 | 替换 SQL 中 {{CONTACT_PHONE}} | 后端运维 | `grep CONTACT_PHONE *.sql` → 无命中 |
| 6 | 执行 init SQL | 后端运维 | 活动数据、配置项插入成功 |
| 7 | 删除 install.php | 后端运维 | `ls public/install.php` → 不存在 |
| 8 | Nginx 配置确认 | 后端运维 | `nginx -t` 通过，server_name 为实际域名 |
| 9 | 后台配置（站点名称/Logo/协议/客服电话） | 产品 | 小程序中验证 |

### 2. 数据库备份

| 序号 | 操作 | 负责人 | 完成标志 |

| 2 | 确认备份文件大小 > 0 | 后端运维 | `ls -lh backup-pre-v1.0.0.sql` |

### 3. 最终预检

| 序号 | 操作 | 负责人 | 完成标志 |
|------|------|-------|---------|
| 1 | 服务器运行 `check-server.sh` | 后端运维 | 全部 PASS |
| 2 | 本地运行 `check-placeholders.sh --strict` | 前端开发 | 退出码 0 |
| 3 | 确认体验版 UAT 全部通过 | 测试 | 无 P0 待修 |

> ⚠️ **决策点**：check-server.sh 有 FAIL → 修复后重新检查，不通过则延期。

## 回退方案

### 场景 1：小程序版本回退

| 步骤 | 操作 |
|------|------|
| 1 | 微信后台 → 版本管理 → 选择上一个版本 → 点击"回退" |
| 2 | 回退后真机验证旧版本功能正常 |
| 3 | 记录回退原因和时间 |

> 注意：微信小程序回退功能仅支持回退到上一个已发布版本。首次发布无旧版本可回退。

### 场景 2：后端代码回退

| 步骤 | 操作 |
|------|------|
| 1 | `cd {{DEPLOY_PATH}}/shopxo-backend` |
| 2 | `git log --oneline -5` 找到上一个稳定版本 commit |
| 3 | `git checkout <commit-hash>` |
| 4 | `nginx -t && systemctl reload nginx` |
| 5 | 真机验证 |

### 场景 3：数据库回退

| 步骤 | 操作 |
|------|------|
| 1 | `mysql -u USER -p DB_NAME < backup-pre-v1.0.0.sql` |
| 2 | 验证数据完整性 |
| 3 | 真机验证 |

> ⚠️ 数据库回退会丢失回退后的所有数据（用户注册、订单、报名等），仅在严重数据错误时使用。

### 场景 4：全量回退（最严重情况）

| 步骤 | 操作 |
|------|------|
| 1 | 小程序版本回退（如有旧版本） |
| 2 | 后端代码回退到上一个稳定版本 |
| 3 | 数据库回退到备份 |
| 4 | Nginx 配置回退 |
| 5 | 全链路验证 |
| 6 | 通知所有相关方 |
