## 4. 权限模型设计

### 4.1 RBAC 权限模型

本系统采用基于角色的访问控制（Role-Based Access Control, RBAC）模型。

**核心关系：**

```
管理员 ──(多对多)──> 角色 ──(多对多)──> 权限
```

**模型说明：**

- 一个管理员可以拥有多个角色
- 一个角色可以包含多个权限
- 权限是系统预定义的最小访问控制单元
- 管理员的最终权限 = 所拥有角色的权限并集

**数据模型：**

| 表名 | 说明 |
|------|------|
| admin | 管理员表，存储账号信息 |
| role | 角色表，存储角色定义 |
| permission | 权限表，存储权限标识 |
| admin_role | 管理员-角色关联表（多对多） |
| role_permission | 角色-权限关联表（多对多） |

**ER关系：**

```
admin ──1:N──> admin_role <──N:1── role ──1:N──> role_permission <──N:1── permission
```

---

### 4.2 权限粒度

本系统权限控制分为三个粒度层次：

#### 4.2.1 模块级权限

控制管理员能否访问某个功能模块。

- 示例：能否访问「商品管理」模块
- 实现：后端路由中间件校验，前端菜单渲染控制
- 判断依据：是否拥有该模块下任意一个权限标识

#### 4.2.2 操作级权限

控制管理员能否执行某个具体操作（增/删/改/查）。

- 示例：能否「创建商品」
- 实现：后端接口权限校验，前端按钮显示/隐藏控制
- 判断依据：是否拥有对应的 `module:action` 权限标识

#### 4.2.3 数据级权限

控制管理员能查看的数据范围。

| 数据范围 | 说明 |
|----------|------|
| 全部数据 | 可查看系统中该模块的所有数据 |
| 本人数据 | 仅可查看自己创建/负责的数据 |

- 示例：普通管理员只能查看自己创建的内容，超级管理员可查看所有内容
- 实现：后端查询时根据管理员角色动态拼接数据过滤条件

---

### 4.3 权限标识命名规范

#### 命名格式

```
module:action
```

- `module`：模块标识，使用小写英文，多单词用下划线连接
- `action`：操作类型，使用小写英文

#### 操作类型定义

| action | 说明 | 对应HTTP方法 |
|--------|------|-------------|
| read | 查看/读取 | GET |
| create | 创建/新增 | POST |
| update | 编辑/修改 | PUT/PATCH |
| delete | 删除 | DELETE |
| export | 导出 | GET（导出接口） |
| audit | 审核 | PUT（审核接口） |

## 7. 供应商角色说明

### 7.1 核心定位

供应商在本系统中**不是系统用户角色**，而是后台内部管理的数据实体。供应商信息仅在后台内部管理，用于商品货源追踪和采购结算。

### 7.2 供应商与系统用户的关系

| 对比项 | 供应商 | 后台管理员 |
|--------|--------|-----------|
| 是否为系统用户 | 否 | 是 |
| 能否登录后台 | 不能 | 能 |
| 是否有账号密码 | 无 | 有 |
| 是否有角色权限 | 无 | 有 |
| 数据存储位置 | supplier表（业务数据） | admin表（系统用户） |

### 7.3 供应商管理规则

| 规则 | 说明 |
|------|------|
| 商品关联 supplier_id | 仅用于内部管理、采购结算和货源追踪 |
| 不提供供应商独立后台 | 供应商无法登录系统 |
| 不允许供应商自行发布商品 | 所有商品由商品管理员在后台统一创建和管理 |
| 不允许供应商自行收款 | 所有订单款项统一进入禧孕文化传媒有限公司账户 |
| 不允许供应商独立处理订单 | 所有订单由订单管理员统一处理 |
| 供应商信息维护 | 由商品管理员在后台维护供应商基本信息 |

### 7.4 供应商数据模型

| 字段 | 类型 | 说明 |
|------|------|------|
| id | bigint | 供应商ID |
| name | varchar(100) | 供应商名称 |
| contact_name | varchar(50) | 联系人姓名 |
| contact_phone | varchar(20) | 联系电话 |
| address | varchar(255) | 供应商地址 |
| status | tinyint | 状态（1启用/0禁用） |
| remark | varchar(500) | 备注 |
| created_at | datetime | 创建时间 |
| updated_at | datetime | 更新时间 |

### 7.5 供应商与商品的关联

```
supplier ──(1:N)──> product
```

- 一个供应商可关联多个商品
- 一个商品仅关联一个供应商
- supplier_id 在商品表中作为外键
- 关联用途：采购成本核算、货源追踪、供应商维度统计

### 8.2 会员等级权益校验

**校验时机：**

| 场景 | 校验内容 | 校验方式 |
|------|----------|----------|
| 优惠券领取 | 会员等级是否满足优惠券领取条件 | 后端校验 coupon.level_limit |
| 专属折扣计算 | 会员等级对应的折扣率 | 后端根据 user.level 计算折扣 |
| 积分计算 | 会员等级对应的积分倍率 | 后端根据 user.level 计算积分 |
| 专属活动参与 | 会员等级是否满足活动参与条件 | 后端校验 activity.level_limit |
| 生日礼发放 | 会员等级是否达到生日礼标准 | 后端定时任务校验 |

**校验流程：**

```
1. 用户发起操作请求
2. 后端从 token 解析用户ID
3. 查询用户会员等级
4. 校验会员等级是否满足操作所需的等级要求
5. 满足则执行操作，不满足则返回权限不足提示
```

**等级判断规则：**

- 会员等级由成长值自动计算，不可手动修改
- 成长值来源：消费金额（1元=1成长值）、签到、评价等
- 等级变更实时生效，降级在每月1日结算时生效

## 9. 后台权限控制规则

### 9.1 登录鉴权

**鉴权流程：**

```
1. 管理员输入账号密码
2. 后端验证账号密码正确性
3. 验证通过后生成 JWT Token
4. Token 包含：管理员ID、角色列表、过期时间
5. 前端存储 Token 并在后续请求中携带
6. 后端中间件统一校验 Token 有效性
```

**Token 规则：**

| 配置项 | 值 |
|--------|-----|
| Token 类型 | JWT (JSON Web Token) |
| 有效期 | 2小时 |
| 刷新机制 | Token 过期前30分钟可刷新 |
| 存储方式 | 前端 localStorage |
| 传输方式 | 请求头 Authorization: Bearer {token} |

**登录安全策略：**

| 策略 | 说明 |
|------|------|
| 密码加密 | bcrypt 哈希存储 |
| 登录失败锁定 | 连续5次失败锁定30分钟 |
| 登录日志 | 记录登录时间、IP地址 |
| 强制下线 | 超级管理员可强制其他管理员下线 |

### 9.2 接口权限校验

**校验流程：**

```
1. 请求到达后端
2. 中间件校验 Token 有效性（未携带或失效返回 401）
3. 从 Token 解析管理员ID
4. 查询管理员的角色列表
5. 查询角色对应的权限标识集合
6. 校验当前接口所需的权限标识是否在权限集合中
7. 有权限则放行，无权限则返回 403
```

**接口权限配置：**

- 每个后端接口标注所需权限标识（通过装饰器/中间件方式）
- 权限标识与接口的映射关系在代码中硬编码，不存数据库
- 示例：

```
GET    /api/admin/products      → 需要 product:read
POST   /api/admin/products      → 需要 product:create
PUT    /api/admin/products/:id  → 需要 product:update
DELETE /api/admin/products/:id  → 需要 product:delete
```

**错误码定义：**

| HTTP状态码 | 错误码 | 说明 |
|-----------|--------|------|
| 401 | UNAUTHORIZED | 未登录或Token失效 |
| 403 | FORBIDDEN | 无权限访问该接口 |

### 9.3 菜单权限控制

**控制规则：**

- 后端根据管理员角色返回有权限的菜单列表
- 前端仅渲染有权限的菜单项
- 无权限的菜单不显示（而非置灰）
- 菜单与权限标识的映射关系在后台配置

**菜单数据结构：**

```json
{
  "id": 1,
  "name": "商品管理",
  "path": "/product",
  "icon": "shopping",
  "sort": 1,
  "permission": "product:read",
  "children": [
    {
      "id": 2,
      "name": "商品列表",
      "path": "/product/list",
      "permission": "product:read"
    }
  ]
}
```

**菜单渲染逻辑：**

```
1. 管理员登录成功
2. 后端根据管理员角色查询有权限的菜单树
3. 前端接收菜单树数据并渲染侧边栏
4. 仅显示有权限的菜单和子菜单
```

### 9.4 按钮权限控制

**控制规则：**

- 页面内的操作按钮（新增、编辑、删除等）根据权限标识控制显示/隐藏
- 无权限的按钮不显示（而非置灰）
- 前端通过权限指令/组件判断按钮是否渲染

**按钮权限标识映射：**

| 按钮类型 | 权限标识后缀 | 示例 |
|----------|-------------|------|
| 新增按钮 | :create | product:create → 新增商品按钮 |
| 编辑按钮 | :update | product:update → 编辑商品按钮 |
| 删除按钮 | :delete | product:delete → 删除商品按钮 |
| 导出按钮 | :export | statistics:export → 导出统计按钮 |
| 审核按钮 | :audit | aftersale:audit → 审核售后按钮 |

**前端实现方式：**

```
方式一：自定义指令
<button v-permission="'product:create'">新增商品</button>

方式二：权限组件
<AuthButton permission="product:create">新增商品</AuthButton>

方式三：权限判断函数
v-if="hasPermission('product:create')"
```

### 9.5 数据权限控制

**控制规则：**

| 数据范围 | 说明 | 适用角色 |
|----------|------|----------|
| 全部数据 | 可查看系统中该模块的所有数据 | 超级管理员 |
| 角色范围数据 | 可查看该角色职责范围内的数据 | 各职能管理员 |
| 只读数据 | 可查看但不可修改 | 财务查看员 |

**数据权限实现：**

```sql
-- 超级管理员：查看所有订单
SELECT * FROM orders WHERE 1=1;

-- 订单管理员：查看所有订单（职责范围内）
SELECT * FROM orders WHERE 1=1;

-- 财务查看员：查看所有订单（只读）
SELECT * FROM orders WHERE 1=1;
-- 但 UPDATE/DELETE 操作被权限标识拦截
```

**数据权限过滤维度：**

| 维度 | 说明 | 示例 |
|------|------|------|
| 模块维度 | 只能查看有权限模块的数据 | 商品管理员无法查看订单数据 |
| 操作维度 | 只能执行有权限的操作 | 财务查看员只能查看不能修改 |
| 行维度 | 只能查看特定条件的数据 | 按组织架构/部门过滤（预留） |

**数据权限校验流程：**

```
1. 管理员发起数据查询/修改请求
2. 后端中间件校验模块级权限（能否访问该模块）
3. 后端中间件校验操作级权限（能否执行该操作）
4. 后端根据管理员角色动态拼接数据过滤条件
5. 执行查询/修改操作
6. 返回结果
```
