# 禧孕母婴用品私域商城小程序 - 订单、支付、售后

## 1. 文档概述

| 项目 | 说明 |
|------|------|
| 项目名称 | 禧孕母婴用品私域商城小程序 |
| 运营方 | 禧孕文化传媒有限公司 |
| 平台类型 | 甲方自营商城（非多商户平台） |
| 支付方式 | 微信支付V3 |
| 文档版本 | v1.0 |

本文件详细描述禧孕母婴用品私域商城小程序中订单、支付、售后三大核心业务模块的设计方案，涵盖完整业务流程、数据结构、接口设计、异常处理及对账机制。所有交易与退款均通过微信支付V3接口完成，商城为自营模式，不存在多商户分账场景。

## 2. 下单流程

### 2.1 下单前置校验

用户在确认订单页点击"提交订单"前，前端与后端需完成以下校验，任何一项不通过则阻止下单并给出明确提示。

| 校验项 | 校验内容 | 不通过提示 | 校验时机 |
|--------|----------|------------|----------|
| 用户登录状态 | 用户已登录且token有效 | "请先登录" | 前端拦截 + 后端校验 |
| 收货地址 | 已选择收货地址且地址完整（省市区+详细地址+收件人+手机号） | "请选择收货地址" | 前端拦截 + 后端校验 |
| 商品状态 | 商品上架且未删除 | "商品已下架" | 后端校验 |
| 库存 | SKU库存充足（可用库存 ≥ 购买数量） | "库存不足" | 后端校验（预扣） |
| 优惠券 | 优惠券有效（未过期、未使用、未锁定）、满足使用条件（满减门槛、适用商品范围） | "优惠券不可用" | 后端校验 |
| 积分 | 积分余额充足、积分未过期 | "积分不足" | 后端校验 |
| 购买限制 | 商品无购买数量限制或未超出限购数量 | "超出限购数量" | 后端校验 |

**校验原则：**
- 前端校验用于快速反馈，提升用户体验
- 后端校验为最终校验，确保数据一致性和安全性
- 所有校验必须原子性执行，避免部分通过部分失败

**各步骤详细说明：**

**步骤1：后端校验商品状态和库存**
- 查询所有下单SKU的当前状态
- 确认商品处于上架状态且未删除
- 确认每个SKU的可用库存 ≥ 购买数量
- 可用库存 = 总库存 - 已售库存 - 预扣库存
- 如有任一SKU校验不通过，整个下单失败

**步骤2：计算订单金额**
- 按照金额计算规则（见2.4节）计算各项金额
- 实付金额最低0.01元
- 如使用优惠券，校验优惠券适用范围与门槛
- 如使用积分，校验积分余额与抵扣上限

**步骤3：预扣库存**
- 将对应SKU的预扣库存增加购买数量
- 使用数据库乐观锁（版本号）或Redis原子操作保证并发安全
- 预扣库存有30分钟有效期，超时自动释放

**步骤4：锁定优惠券**
- 将优惠券状态从"未使用"变更为"锁定"
- 记录锁定关联的订单ID
- 锁定状态优惠券不可被其他订单使用

**步骤5：扣减积分**
- 从用户积分余额中扣减使用的积分数
- 记录积分变动日志（类型：积分抵扣，关联订单号）
- 积分扣减使用数据库行锁保证并发安全

**步骤6：生成订单号**
- 按照订单号生成规则（见2.3节）生成唯一订单号
- 数据库唯一索引保证唯一性

**步骤7：创建订单记录**
- 写入订单主表，包含订单号、用户ID、金额信息、收货地址、优惠券ID、使用积分等
- 订单初始状态为"待付款"

**步骤8：创建订单商品记录**
- 写入订单商品表，每条记录对应一个SKU
- 包含商品ID、SKU ID、商品名称、SKU属性、单价、数量、小计金额
- 快照商品下单时的信息，后续商品修改不影响订单

**步骤9：创建支付记录**
- 写入支付记录表
- 包含关联订单号、支付金额、支付方式（微信支付）、支付状态（待支付）
- 用于后续支付流程和退款流程

**步骤10：清除购物车已下单商品**
- 从用户购物车中删除已下单的商品
- 按SKU维度匹配删除

**步骤11：返回订单ID和支付参数**
- 返回订单ID、订单号
- 同时调用微信支付统一下单API，返回前端支付所需参数（见3.1节）

**事务处理：**
- 步骤3~10必须在同一数据库事务中执行
- 事务失败时回滚所有操作（库存预扣、优惠券锁定、积分扣减、订单创建等）
- 微信支付统一下单调用在事务提交后执行，避免事务回滚但支付已创建

### 2.3 订单号生成规则

| 规则项 | 说明 |
|--------|------|
| 格式 | `XY` + `yyyyMMddHHmmss` + 6位随机数 |
| 示例 | `XY20260520143025837261` |
| 前缀 | `XY` 代表"禧孕" |
| 时间部分 | 14位，精确到秒 |
| 随机部分 | 6位数字，范围000000~999999 |
| 总长度 | 22位 |
| 唯一性保证 | 数据库订单号字段设置唯一索引；生成时若冲突则重新生成（最多3次） |

**生成算法：**

```
function generateOrderNo(): string {
    const prefix = 'XY';
    const timestamp = formatDate(new Date(), 'yyyyMMddHHmmss');
    const random = Math.floor(Math.random() * 1000000).toString().padStart(6, '0');
    return prefix + timestamp + random;
}
```

**冲突处理：**
- 插入数据库时若触发唯一索引冲突，重新生成订单号
- 最多重试3次
- 3次仍冲突则返回系统繁忙提示

### 2.4 金额计算详细规则

#### 2.4.1 金额计算公式

```
商品总金额 = Σ(SKU单价 × 数量)
活动优惠   = Σ(原价 - 活动价) × 数量    （无活动时为0）
优惠券抵扣 = 优惠券面值                  （未使用时为0）
积分抵扣   = 使用积分数 ÷ 100            （未使用时为0）
运费       = 运费模板计算结果             （满包邮时为0）

实付金额 = 商品总金额 - 活动优惠 - 优惠券抵扣 - 积分抵扣 + 运费
```

#### 2.4.2 各项金额详细说明

**商品总金额**
- 以SKU下单时的单价为准
- 单价取商品表中的销售价格
- 不含任何优惠的原始金额

**活动优惠**
- 当商品参与促销活动时，活动价与原价的差额
- 活动优惠按SKU维度计算
- 同一SKU只享受最优惠的一个活动
- 活动优惠不可与优惠券叠加时，取优惠力度更大者

**优惠券抵扣**
- 满减券：订单金额满足满减门槛时，抵扣券面值
- 折扣券：按折扣比例计算抵扣金额
- 优惠券抵扣金额不超过商品总金额减去活动优惠后的金额
- 优惠券有适用商品范围限制时，仅计算适用商品的金额

**积分抵扣**
- 积分抵扣比例：100积分 = 1元
- 积分抵扣上限：不超过商品总金额减去活动优惠和优惠券抵扣后的30%
- 单笔订单积分抵扣上限可后台配置
- 积分抵扣后实付金额不能为0

**运费**
- 根据运费模板计算
- 运费模板规则：按件数/按重量/按体积
- 满足满包邮条件时运费为0
- 不同收货地区可能有不同运费

#### 2.4.3 金额计算示例

**示例1：普通购买**

| 项目 | 金额 |
|------|------|
| 婴儿连体衣 × 2 | ¥198.00 |
| 奶粉罐 × 1 | ¥298.00 |
| 商品总金额 | ¥496.00 |
| 活动优惠 | ¥0.00 |
| 优惠券抵扣（满400减50） | ¥50.00 |
| 积分抵扣（500积分） | ¥5.00 |
| 运费 | ¥0.00（满包邮） |
| **实付金额** | **¥441.00** |

**示例2：活动商品 + 积分抵扣**

| 项目 | 金额 |
|------|------|
| 婴儿推车 × 1（原价¥1299，活动价¥999） | ¥999.00 |
| 商品总金额 | ¥1,299.00 |
| 活动优惠 | ¥300.00 |
| 优惠券抵扣 | ¥0.00（未使用） |
| 积分抵扣（2000积分） | ¥20.00 |
| 运费 | ¥0.00（大件包邮） |
| **实付金额** | **¥979.00** |

#### 2.4.4 金额精度与舍入规则

- 所有金额以"分"为单位存储，避免浮点精度问题
- 展示时转换为"元"，保留2位小数
- 计算过程中间值保留4位小数，最终结果四舍五入到分
- 实付金额最低0.01元（1分），若计算结果低于0.01元则取0.01元

## 3. 微信支付流程

### 3.1 支付参数生成

订单创建成功后，后端调用微信支付V3统一下单API，生成前端调起支付所需的全部参数。

**详细步骤：**

```

      │
      ▼

│    - appid                   │
│    - mchid                   │
│    - description（商品描述）   │
│    - out_trade_no（订单号）    │
│    - notify_url（回调地址）    │
│    - amount.total（金额-分）   │
│    - amount.currency（CNY）   │
└──────────────┬──────────────┘
               │
               ▼
┌─────────────────────────────┐
│ 2. 调用微信支付V3统一下单API   │
│    POST /v3/pay/transactions/jsapi │
│    使用商户私钥签名            │
└──────────────┬──────────────┘
               │ 成功
               ▼
┌─────────────────────────────┐
│ 3. 获取prepay_id             │
│    预支付交易会话标识          │
└──────────────┬──────────────┘
               │
               ▼

└─────────────────────────────┘
```

**统一下单请求参数：**

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| appid | string | 是 | 小程序AppID |
| mchid | string | 是 | 商户号 |
| description | string | 是 | 商品描述，最长127字节，格式：禧孕商城-商品名 |
| out_trade_no | string | 是 | 商户订单号，即系统订单号 |
| notify_url | string | 是 | 支付回调通知地址 |
| amount.total | int | 是 | 订单总金额，单位为分 |
| amount.currency | string | 是 | 货币类型，固定CNY |
| payer.openid | string | 是 | 用户openid |

**签名参数生成：**

```
签名串 = appId + "\n" + timeStamp + "\n" + nonceStr + "\n" + prepay_id + "\n"
paySign = SHA256withRSA(签名串, 商户私钥)
```

### 3.2 前端支付调用

前端收到后端返回的支付参数后，调用微信小程序支付接口 `wx.requestPayment`。

**调用代码示例：**

```javascript
wx.requestPayment({
  timeStamp: payParams.timeStamp,
  nonceStr: payParams.nonceStr,
  package: payParams.package,
  signType: 'RSA',
  paySign: payParams.paySign,
  success(res) {
    // 支付成功，跳转到支付结果页
    // 注意：以此回调作为支付成功的展示依据
    // 实际支付状态以后端回调为准
    wx.navigateTo({ url: '/pages/order/pay-result?orderId=' + orderId })
  },
  fail(err) {
    if (err.errMsg === 'requestPayment:fail cancel') {
      // 用户取消支付，订单保持待付款状态
      wx.showToast({ title: '已取消支付', icon: 'none' })
    } else {
      // 支付失败，提示用户
      wx.showToast({ title: '支付失败，请重试', icon: 'none' })
    }
  }
})
```

**支付参数说明：**

| 参数 | 说明 |
|------|------|
| timeStamp | 当前时间戳（秒级） |
| nonceStr | 随机字符串 |
| package | 格式为 `prepay_id=wx...` |
| signType | 签名类型，固定为 `RSA` |
| paySign | 签名值 |

**前端注意事项：**
- `wx.requestPayment` 的 success 回调仅表示用户侧支付流程完成，不代表支付真正成功
- 支付结果页需调用后端接口查询订单支付状态
- 避免在 success 回调中直接更新本地订单状态

### 3.3 微信支付回调处理

微信支付成功后，会向商户后台发送支付结果通知。后端需正确处理回调，确保订单状态与支付状态一致。

**回调数据解密说明：**

微信支付V3回调数据使用 AES-256-GCM 加密，解密需要以下参数：
- `algorithm`：AEAD_AES_256_GCM
- `nonce`：回调数据中的 nonce 字段
- `associated_data`：回调数据中的 associated_data 字段
- `ciphertext`：回调数据中的 ciphertext 字段（Base64编码）
- `key`：APIv3密钥（32字节）

**回调响应规范：**
- 处理成功：HTTP 200，返回 `{"code":"SUCCESS","message":"成功"}`
- 处理失败：HTTP 200，返回 `{"code":"FAIL","message":"失败原因"}`
- 微信会对未成功通知的回调进行重试，重试间隔为15s/15s/30s/3m/10m/20m/30m/30m/30m/60m/3h/3h/3h/6h/6h，总计约24小时

**支付成功幂等规则：**

- processPaymentSuccess 必须基于数据库当前订单状态判断，而不是只依赖调用方
- 如果订单已处于 pending_delivery/delivered/completed 状态，则直接幂等返回，不重复处理
- 如果订单不是 pending_payment 且不是已支付状态，记录异常日志并拒绝处理
- 支付回调中的金额必须与订单实付金额一致（精确到分），不一致则拒绝处理并告警

### 3.4 支付失败处理

| 场景 | 处理方式 | 订单状态 |
|------|----------|----------|
| 用户取消支付 | 订单保持待付款状态，等待用户重新支付或超时自动关闭 | 待付款 |
| 支付超时 | 订单创建30分钟后未支付，自动关闭（见第4节） | 已取消 |
| 网络异常 | 前端调用后端查询支付结果接口，以查询结果为准 | 视查询结果 |
| 余额不足 | 微信支付侧提示用户，订单保持待付款 | 待付款 |
| 微信系统异常 | 后端记录异常日志，订单保持待付款，等待回调或用户重试 | 待付款 |
| 重复支付 | 幂等处理：检测到订单已支付，不再重复处理，原路退回重复支付金额 | 已付款 |

**网络异常处理详细流程：**

```
wx.requestPayment 调用后网络异常
        │
        ▼
前端弹出提示"支付结果确认中"
        │
        ▼
前端轮询后端查询支付结果接口（间隔3秒，最多10次）
        │
        ├── 查询到已支付 → 跳转支付成功页
        ├── 查询到未支付 → 提示用户重新支付
        └── 超过轮询次数 → 提示用户稍后在订单列表查看
```

**重复支付幂等处理：**
- 支付回调处理前先查询订单状态
- 若订单已支付，忽略本次回调，返回成功响应给微信
- 若检测到同一订单存在多笔支付记录，对重复支付金额发起退款
- 退款前记录异常日志，通知管理员

### 3.5 支付安全

| 安全措施 | 说明 |
|----------|------|
| 金额校验 | 支付回调中的金额必须与订单实付金额一致（精确到分），不一致则拒绝处理并告警 |
| 防重复通知 | 支付回调处理采用幂等设计，同一笔支付通知多次处理结果一致 |
| 签名验证 | 验证微信回调签名，确保通知来源可信；验证时间戳防止重放攻击 |
| HTTPS传输 | 所有与微信支付API的通信均使用HTTPS |
| 证书管理 | 定期更新微信平台证书，处理证书轮换 |
| 敏感信息 | 商户私钥、APIv3密钥等敏感信息加密存储，不记录到日志 |
| 请求防篡改 | 前端不传递金额，金额由后端根据订单计算，防止金额篡改 |
| 防并发 | 订单支付状态更新使用数据库乐观锁或分布式锁，防止并发更新 |

## 4. 订单超时关闭

### 4.1 超时关闭机制

| 配置项 | 值 | 说明 |
|--------|-----|------|
| 超时时间 | 30分钟 | 订单创建后30分钟未支付则自动关闭 |
| 实现方式 | 定时任务扫描 + Redis延迟队列 | 双重保障 |
| 扫描频率 | 每分钟一次 | 定时任务 |
| 关闭后操作 | 归还库存、归还优惠券、归还积分 | 资源释放 |
| 通知 | 微信订阅消息 + 站内消息 | 通知用户订单已关闭 |

**双重保障机制说明：**
- **Redis延迟队列**（主）：订单创建时将订单ID加入Redis延迟队列，30分钟后消费消息触发关闭
- **定时任务扫描**（备）：每分钟扫描超过30分钟未支付的订单，兜底处理Redis延迟队列遗漏的订单
- 两种机制均保证幂等，不会重复关闭

### 4.2 超时关闭流程

```
定时任务触发 / Redis延迟队列消费
        │
        ▼
┌─────────────────────────────┐
│ 1. 查询超时未支付订单          │
│    条件：                      │
│    - 订单状态 = 待付款          │
│    - 创建时间 + 30分钟 < 当前时间│
└──────────────┬──────────────┘
               │
               ▼
┌─────────────────────────────┐
│ 2. 幂等检查                   │
│    - 再次确认订单状态为待付款    │
│    - 使用乐观锁更新状态         │
│    - 已非待付款则跳过           │
└──────────────┬──────────────┘
               │
               ▼
┌─────────────────────────────┐
│ 3. 更新订单状态为已取消         │
│    - 订单状态 → 已取消          │
│    - 取消原因 → 超时未支付      │
│    - 取消时间 → 当前时间        │
└──────────────┬──────────────┘
               │
               ▼
┌─────────────────────────────┐
│ 4. 归还预扣库存               │
│    - 预扣库存减少对应数量       │
│    - 可用库存恢复              │
└──────────────┬──────────────┘
               │
               ▼
┌─────────────────────────────┐
│ 5. 解锁优惠券                 │
│    - 优惠券状态：锁定 → 未使用  │
│    - 清除关联订单ID            │
└──────────────┬──────────────┘
               │
               ▼
┌─────────────────────────────┐
│ 6. 归还积分                   │
│    - 用户积分余额增加          │
│    - 记录积分变动日志          │
│    - 类型：超时关闭归还        │
└──────────────┬──────────────┘
               │
               ▼
┌─────────────────────────────┐
│ 7. 记录订单日志               │
│    - 操作：超时自动关闭        │
│    - 操作者：系统              │
│    - 变更：待付款 → 已取消     │
└──────────────┬──────────────┘
               │
               ▼
┌─────────────────────────────┐
│ 8. 通知用户                   │
│    - 微信订阅消息              │
│    - 站内消息                 │
└─────────────────────────────┘
```

**注意事项：**
- 步骤3~6在同一数据库事务中执行
- 若事务执行失败，定时任务下次扫描时会重试
- 归还库存时需校验商品是否仍存在，避免归还到已删除商品
- 优惠券归还时需校验优惠券是否仍在有效期内，已过期的优惠券不再归还可用状态

## 5. 发货流程

### 5.1 发货操作

**发货操作步骤：**

1. 管理员进入"订单管理 → 待发货"列表
2. 选择待发货订单，点击"发货"
3. 填写发货信息：

**发货校验：**
- 订单状态必须为"已付款"
- 物流公司和物流单号为必填项

**物流信息获取方式：**
- 对接第三方物流查询API（如快递100、快递鸟等）

- 物流状态变更时通过回调更新

**物流状态映射：**

| 物流状态 | 说明 | 展示文案 |
|----------|------|----------|

- 物流状态为"已签收"时，不再查询
- 物流状态为"异常"时，通知管理员

## 7. 售后申请

### 7.1 售后申请条件

| 条件 | 说明 |
|------|------|
| 订单状态 | 已付款 / 已发货 / 已完成 |
| 时间限制 | 确认收货后7天内可申请（已付款/已发货状态无时间限制） |
| 次数限制 | 每个订单项（SKU维度）只能申请一次售后 |
| 金额限制 | 售后金额不能大于该订单项的实付金额 |
| 特殊限制 | 已申请售后的订单项不可重复申请 |

**不可申请售后的情况：**
- 订单状态为待付款、已取消
- 超过售后申请时效
- 该订单项已有进行中/已完成的售后单
- 虚拟商品/特殊商品（后台标记不可售后）

### 7.2 售后类型

| 售后类型 | 适用订单状态 | 说明 |
|----------|-------------|------|
| 仅退款 | 已付款（未发货） | 订单已付款但尚未发货，申请全额或部分退款 |
| 退货退款 | 已发货 / 已完成 | 已收到商品，需退回商品后退款 |

**售后类型选择规则：**
- 订单状态为"已付款"时，仅可选择"仅退款"
- 订单状态为"已发货"或"已完成"时，仅可选择"退货退款"
- 前端根据订单状态自动限定可选类型

**微信退款API请求参数：**

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| out_trade_no | string | 是 | 原支付订单号 |
| out_refund_no | string | 是 | 退款单号 |
| reason | string | 否 | 退款原因 |
| amount.refund | int | 是 | 退款金额，单位分 |
| amount.total | int | 是 | 原订单金额，单位分 |
| amount.currency | string | 是 | 货币类型，CNY |
| notify_url | string | 是 | 退款回调通知地址 |

**积分扣回规则：**
- 退款金额占实付金额的比例 × 确认收货时发放的积分数 = 需扣回积分数
- 若用户当前积分不足扣回，则扣回全部可用积分
- 积分扣回记录日志

**退款发起流程：**

1. 管理员点击退款后，售后单状态进入 pending_refund
2. 检查同一售后单是否已有 pending/processing/success 的退款单
   - 已有 pending/processing 时，返回已有退款单信息，不重复向微信发起退款
   - 已有 success 时，拒绝重复退款
3. 检查同一订单累计退款金额：所有 success/pending 退款金额之和 + 本次退款金额 <= order.payAmount
4. 调用微信退款 API
   - 成功：创建 OrderRefund 记录，状态为 pending
   - 失败：不创建 OrderRefund 记录，抛出异常

**退款回调幂等规则：**

- 重复 SUCCESS 回调不会重复增加库存、不会重复归还/扣回积分
- 通过 OrderRefund.status 判断：已 success 的退款单不再处理
- 退款失败（CLOSED/ABNORMAL）时，售后单保持 pending_refund 状态，不直接关闭售后单
- 退款失败时记录日志，等待管理员检查后重试

**退款状态枚举：**

| 状态值 | 说明 |
|--------|------|
| initiating | 本地已创建退款意图，尚未完成微信退款请求 |
| pending | 微信已受理，等待回调 |
| processing | 本地正在处理回调副作用 |
| success | 退款成功（所有本地副作用已完成） |
| failed | 退款失败 |
| closed | 退款关闭 |
| abnormal | 退款异常 |

**售后状态流转（含退款）：**

待审核(pending_review) → 审核通过(approved) / 审核拒绝(rejected)
审核通过(approved) → 已退货(returned) [退货退款场景]
已退货(returned) → 待退款(pending_refund) [管理员确认退款]
审核通过(approved) → 待退款(pending_refund) [仅退款场景]
待退款(pending_refund) → 已退款(refunded) [微信退款 SUCCESS 回调]
待退款(pending_refund) → 待退款(pending_refund) [微信退款失败，保持状态，记录日志]

## 10. 后台审核流程

### 10.1 审核操作

| 操作 | 说明 | 售后状态变更 |
|------|------|-------------|
| 审核通过 | 同意用户的售后申请 | 待审核 → 审核通过 |
| 审核拒绝 | 拒绝用户的售后申请，需填写拒绝原因 | 待审核 → 审核拒绝 |
| 标记处理中 | 售后单正在处理中（如与用户沟通中） | 待审核 → 处理中 |

**审核操作界面要素：**

- 售后单基本信息（售后单号、申请时间、售后类型、售后原因）
- 用户信息（昵称、手机号、会员等级）
- 订单信息（订单号、下单时间、实付金额、订单状态）
- 商品信息（商品名称、SKU属性、单价、数量、实付金额）
- 售后详情（问题描述、凭证图片、退款金额）
- 用户历史售后记录
- 审核操作区域（通过/拒绝/处理中）
- 备注区域（管理员内部备注，用户不可见）

### 10.2 审核注意事项

**查看用户历史售后记录：**
- 展示用户近6个月的售后申请记录
- 包括售后类型、售后原因、处理结果
- 用于判断用户是否存在恶意售后行为
- 高频售后用户（月售后率>50%）标记提醒

**查看订单信息：**
- 订单完整信息，包括所有订单项
- 订单支付信息（支付时间、支付金额）
- 订单物流信息（物流公司、物流单号、物流状态）
- 订单操作日志

**查看凭证图片：**
- 支持放大查看
- 支持下载原图
- 多图支持左右滑动

**记录审核意见：**
- 审核通过：可选填写审核备注
- 审核拒绝：必填拒绝原因，拒绝原因将展示给用户
- 管理员内部备注：仅后台可见，用于内部沟通

## 11. 退款记录

### 11.1 退款信息

每笔退款操作均创建退款记录，完整记录退款全过程。

| 字段 | 类型 | 说明 |

| fail_reason | string | 失败原因 |
| create_time | datetime | 创建时间 |
| update_time | datetime | 更新时间 |

**退款单号生成规则：**
- 格式：`RF` + `yyyyMMddHHmmss` + 6位随机数
- 示例：`RF20260520143025837261`
- 唯一性保证：数据库唯一索引

### 11.2 退款状态

| 状态值 | 状态名 | 说明 |
|--------|--------|------|
| 0 | 退款中 | 已调用微信退款API，等待退款结果 |
| 1 | 退款成功 | 微信退款回调确认退款成功 |

**退款状态流转：**

```
退款中 ──→ 退款成功
  │
  └──→ 退款失败 ──→ 重试 ──→ 退款中
```

### 11.3 退款失败处理

**失败原因分类：**

| 失败原因 | 处理方式 |
|----------|----------|
| 商户余额不足 | 充值商户账户后重试 |
| 退款单号重复 | 检查是否已退款成功，避免重复退款 |
| 订单已全额退款 | 检查退款金额是否超出可退金额 |
| 微信系统异常 | 稍后重试 |
| 账户异常 | 联系微信支付客服 |

**重试机制：**
- 退款失败后自动重试，最多3次
- 重试间隔：5分钟、30分钟、2小时
- 重试时使用原退款单号（微信退款支持幂等）
- 3次重试仍失败，标记为"需人工处理"
- 通知管理员人工介入

**人工处理：**
- 管理员查看退款失败详情
- 可手动触发重试
- 可标记为"已人工处理"
- 记录人工处理备注

## 13. 异常场景处理

### 13.1 支付成功但回调未收到

**场景描述：** 用户实际已支付成功，但微信支付回调通知未到达商户服务器（网络异常、服务器宕机等）。

**处理方案：**

| 方案 | 触发时机 | 处理方式 |
|------|----------|----------|
| 前端主动查询 | 用户支付后跳转支付结果页 | 前端调用后端查询支付结果接口，后端调用微信支付查询API获取支付状态 |
| 定时任务对账 | 每小时 | 扫描"待付款"状态超过5分钟的订单，调用微信支付查询API确认支付状态 |
| 人工对账 | 每日 | 导出微信支付账单与系统订单对账，发现差异人工处理 |

**定时任务对账流程：**

```
每小时执行
      │
      ▼
查询条件：订单状态=待付款 且 创建时间在5分钟~2小时之间
      │
      ▼
逐单调用微信支付查询API
GET /v3/pay/transactions/out-trade-no/{out_trade_no}
      │
      ├── 返回SUCCESS → 更新订单为已支付，执行支付成功后处理
      ├── 返回NOTPAY → 保持待付款，等待用户支付或超时关闭
      └── 返回CLOSED → 更新订单为已取消
```

### 13.2 退款超时

**场景描述：** 调用微信退款API后，长时间未收到退款回调通知。

**处理方案：**

| 方案 | 触发时机 | 处理方式 |
|------|----------|----------|
| 主动查询退款状态 | 调用退款API后30分钟未收到回调 | 调用微信退款查询API |
| 定时任务扫描 | 每小时 | 扫描"退款中"状态超过30分钟的退款记录，查询退款状态 |
| 重试机制 | 查询确认退款失败 | 按重试策略重新发起退款 |

**退款查询API：**

```
GET /v3/refund/domestic/refunds/{out_refund_no}
```

**退款状态映射：**

| 微信退款状态 | 系统处理 |
|-------------|----------|
| PROCESSING | 保持退款中，继续等待 |
| SUCCESS | 更新为退款成功 |
| CHANGE | 退款异常，退款到银行发现用户的卡作废或者冻结了，导致原路退款银行卡失败，资金回流到用户的商户号，退款状态为"退款异常" |
| REFUNDCLOSE | 退款关闭，更新为退款失败 |

### 13.3 重复退款

**场景描述：** 同一笔售后单被多次发起退款，导致重复退款。

**防护措施：**

| 措施 | 说明 |
|------|------|
| 售后单状态控制 | 售后单进入退款流程后状态变更，不可重复发起退款 |
| 退款单号唯一 | 每笔退款使用唯一退款单号，微信退款API支持幂等 |
| 退款前校验 | 发起退款前校验：售后单状态是否允许退款、退款金额是否超出可退金额 |
| 数据库约束 | 退款记录表对售后单号+退款金额设置约束，防止重复插入 |
| 对账校验 | 每日对账时检查是否存在重复退款 |

**退款前校验逻辑：**

```
1. 查询售后单状态，确认处于"可退款"状态
2. 查询该售后单是否已有成功的退款记录
3. 计算可退金额 = 订单实付金额 - 已退款金额
4. 本次退款金额 ≤ 可退金额
5. 以上校验全部通过才允许发起退款
```

### 13.4 库存异常

**场景1：预扣后商品被下架**

| 项目 | 说明 |
|------|------|
| 场景 | 用户下单预扣库存后，管理员将商品下架 |
| 影响 | 用户支付后商品已下架 |
| 处理 | 支付成功后不阻断，正常发货；若无法发货，管理员主动联系用户退款 |

**场景2：预扣后库存被调整**

| 项目 | 说明 |
|------|------|
| 场景 | 管理员在后台调整库存，将库存减少到低于预扣数量 |
| 影响 | 可能出现超卖 |
| 处理 | 库存调整时校验：新库存值 ≥ 已售库存 + 预扣库存；若不满足则提示管理员 |

**场景3：并发下单导致超卖**

| 项目 | 说明 |
|------|------|
| 场景 | 多个用户同时下单同一SKU，库存不足 |
| 处理 | 预扣库存使用Redis原子操作（DECRBY），库存不足时下单失败 |

**库存安全策略：**

```
预扣库存使用Redis原子操作：
1. available_stock = Redis.GET(sku:stock:available)
2. if available_stock < quantity: return 库存不足
3. result = Redis.DECRBY(sku:stock:available, quantity)
4. if result < 0:
     Redis.INCRBY(sku:stock:available, quantity)  // 回滚
     return 库存不足
5. return 预扣成功
```

## 16. 资源锁定与释放规则

### 16.1 优惠券状态流转

| 状态值 | 状态名 | 说明 |
|--------|--------|------|
| 1 | 未使用(FREE) | 优惠券可用，可被选择使用 |
| 2 | 锁定中(LOCKED) | 下单时锁定，不可被其他订单使用 |
| 3 | 已使用(USED) | 支付成功后标记，不可恢复 |
| 4 | 已过期(EXPIRED) | 超过有效期，不可使用 |

**状态流转规则：**

| 触发事件 | 状态变更 | 说明 |
|----------|----------|------|
| 下单 | FREE → LOCKED | 在订单创建事务中锁定 |
| 支付成功 | LOCKED → USED | 在支付成功事务中标记已使用 |
| 未支付取消 | LOCKED → FREE | 仅释放 LOCKED 状态的优惠券 |
| 超时关闭 | LOCKED → FREE | 仅释放 LOCKED 状态的优惠券 |
| 退款成功 | 不退回 | 已使用优惠券按业务规则不退回 |

**关键约束：**
- 取消/超时关闭时只释放 LOCKED 状态的优惠券，USED 状态不释放
- 已使用优惠券退款后不退回（按业务规则）
- 下单时校验优惠券必须为 FREE 状态

### 16.2 库存扣减与释放

| 触发事件 | 操作 | 说明 |
|----------|------|------|
| 下单 | 条件扣减 `stock >= quantity` | 使用 updateMany 原子操作，失败回滚整个订单 |
| 未支付取消 | 归还库存 `stock += quantity` | 在取消事务中执行 |
| 超时关闭 | 归还库存 `stock += quantity` | 在关闭事务中执行 |
| 支付成功 | 不操作 | 库存已在下单时扣减，支付后无需再操作 |
| 售后退款 | 归还库存 `stock += quantity` | 仅退货退款(type=2)时归还 |

**并发安全：**
- 下单扣减使用 `updateMany where stock >= quantity`，保证不会超卖
- 扣减失败（count=0）时回滚整个订单创建事务
- 写入 productStockLog 记录每次库存变动

### 16.3 积分抵扣与返还

| 触发事件 | 操作 | 说明 |
|----------|------|------|
| 下单抵扣 | 扣减 availablePoints | 在订单创建事务中扣减，写入 pointsRecord |
| 未支付取消 | 返还抵扣积分 | 在取消事务中 increment |
| 超时关闭 | 返还抵扣积分 | 在关闭事务中 increment |
| 支付成功 | 不重复处理 | 抵扣积分已在下单时扣减 |
| 确认收货 | 发放奖励积分 | 按实付金额计算，1元=1积分 |
| 售后退款 | 扣回奖励积分 + 按比例返还抵扣积分 | 在退款成功事务中处理 |

**积分返还规则：**
- 抵扣积分返还：取消/超时时全额返还；退款时按退款金额占实付金额比例返还
- 奖励积分扣回：退款金额占实付金额比例 × 确认收货时发放的积分数
- 用户积分不足扣回时，扣回全部可用积分

### 16.4 退款发起流程（initiating 模式）

1. 管理员点击退款后，先创建 `status=initiating` 的 OrderRefund 记录
2. 调用微信退款 API
3. 微信失败 → 更新 OrderRefund 为 `failed`，不修改 AftersaleOrder
4. 微信成功 → 事务内更新 OrderRefund 为 `pending`，AftersaleOrder 为 `pending_refund`
5. 本地事务失败 → 保留 `initiating`，可通过 admin sync 接口修复

### 16.5 退款回调一致性

- 退款 SUCCESS 回调在单个事务中完成：`pending/initiating → processing → 副作用 → success`
- `orderRefund.status=success` 代表所有本地副作用（库存归还、积分处理、售后单更新）已完成
- 不存在"退款记录 success 但售后单/库存/积分未处理"的窗口
- 重复回调不会重复副作用
- 孤儿回调（找不到 OrderRefund）返回 FAIL 并写入 RefundCallbackLog
