# 邀请裂变逻辑说明

## 一、一级邀请关系设计

### 1.1 设计原则

- **仅一级邀请**：无多级链路，不涉及二级/三级分销
- 邀请人 → 被邀请人，一对一关系
- 一个被邀请人只能有一个邀请人
- 一个邀请人可以邀请多个被邀请人

### 1.2 关系存储

邀请关系通过 `sxo_invite_reward` 表隐式存储，不单独建邀请关系表：

- 每条 `trigger_event=register` 的记录代表一条邀请关系
- `inviter_id`：邀请人
- `invitee_id`：被邀请人
- 查询某用户的所有被邀请人：`SELECT DISTINCT invitee_id FROM sxo_invite_reward WHERE inviter_id = ?`

## 二、邀请码生成逻辑

### 2.1 生成算法

```php
$invite_code = md5($user_id . 'muying_invite');
```

- 输入：用户ID + 固定盐值 `muying_invite`
- 输出：32位MD5哈希字符串
- 每个用户ID对应唯一邀请码

### 2.2 邀请码反查

```php
private static function GetInviterByCode($code)
{
    $users = Db::name('User')->column('id', 'id');
    foreach ($users as $uid) {
        if (md5($uid . 'muying_invite') === $code) {
            return $uid;
        }
    }
    return 0;
}
```

**性能说明**：当前实现遍历所有用户做MD5比对，用户量大时性能较差。MVP阶段可接受，后续优化方向：
- 新增 `sxo_user.invite_code` 字段，注册时生成并存储
- 反查时直接 `WHERE invite_code = ?`，O(1)查询

### 2.3 邀请码使用场景

- 注册时：URL参数 `?invite_code=xxx`，前端 `pages/login/login.vue` 支持 `invite_code` 参数
- 邀请页面：用户在 `pages/invite/invite.vue` 复制邀请码分享给好友
- 邀请海报：`InviteService::Poster()` 返回邀请码和邀请链接

## 三、注册钩子

### 3.1 触发时机

在 `UserService::Reg()` 方法中，用户注册成功后调用：

```php
// 文件：app/service/UserService.php 约第1654行
InviteService::OnUserRegister([
    'user_id'     => $user_id,
    'invite_code' => isset($params['invite_code']) ? $params['invite_code'] : '',
]);
```

### 3.2 OnUserRegister 处理逻辑

```php
public static function OnUserRegister($params = [])
```

**执行流程**：

1. 校验 `user_id` 不为空
2. 获取 `invite_code`，为空则返回（无邀请码=自然注册，不触发奖励）
3. 通过 `GetInviterByCode()` 反查邀请人ID
4. 校验邀请码有效性
5. 校验不能邀请自己（inviter_id != user_id）
6. 插入 `sxo_invite_reward` 记录：
   - `reward_type = 'integral'`
   - `reward_value = 0`（注册仅绑定关系，不发放积分）
   - `trigger_event = 'register'`
   - `status = 1`（GRANTED，已发放——因为奖励值为0，无需待发放）
7. 唯一约束兜底：如果应用层幂等校验未拦截，数据库唯一索引会拦截重复插入

**重要说明**：注册钩子仅绑定邀请关系，不发放任何积分。首单奖励是唯一的奖励触发点。

### 3.3 注册记录说明

| 字段 | 值 | 说明 |
|------|-----|------|
| reward_value | 0 | 注册不发放积分 |
| status | 1 (GRANTED) | 奖励值为0时直接标记为已发放 |
| trigger_event | 'register' | 仅作为邀请关系绑定记录 |

**配置方式**：注册奖励已统一移除，后台 `InviteConfig` 页面会强制将 `muying_invite_register_reward` 和 `muying_invite_register_auto_grant` 置为 0。

## 四、首单钩子

### 4.1 触发时机

在 `OrderService` 中，订单支付成功后调用：

```php
// 文件：app/service/OrderService.php 约第1021行
InviteService::OnFirstOrder(['user_id' => $order['user_id']]);
```

### 4.2 OnFirstOrder 处理逻辑

```php
public static function OnFirstOrder($params = [])
```

**执行流程**：

1. 校验 `user_id` 不为空
2. 校验订单状态（如提供 order_id，异常订单不触发）
3. 查询该用户是否有 `trigger_event=register` 的邀请记录
4. 如果无邀请记录，返回（非被邀请用户，不触发首单奖励）
5. 获取邀请人ID（从注册邀请记录的 `inviter_id`）
6. 幂等校验：查询是否已存在 `inviter_id + invitee_id + trigger_event=first_order` 的记录
7. 每日上限检查：调用 `CheckDailyLimit($inviter_id)`，超限则不创建记录
8. 读取首单奖励配置：`MyC('muying_invite_first_order_reward', 0, true)`
9. 读取自动发放配置：`MyC('muying_invite_first_order_auto_grant', 1, true)`
10. 插入 `sxo_invite_reward` 记录：
    - `reward_type = 'integral'`
    - `reward_value = 配置值`
    - `trigger_event = 'first_order'`
    - `status = 0`（PENDING，待发放）
11. 如果 `reward_value > 0` 且 `auto_grant = 1`，立即调用 `GrantRewardInner()` 发放积分
12. 唯一约束兜底：数据库唯一索引拦截重复插入

### 4.3 首单奖励配置

| 配置键 | 默认值 | 说明 |
|--------|--------|------|
| muying_invite_first_order_reward | 0 | 首单奖励积分数，0表示不发放 |
| muying_invite_first_order_auto_grant | 1 | 是否自动发放（1=自动，0=需后台手动补发） |
| muying_invite_daily_limit | 0 | 每日奖励上限（0=不限制） |

## 五、奖励发放逻辑

### 5.1 优先级

当前MVP阶段仅支持积分奖励，奖励类型 `reward_type` 预留了 `coupon`（优惠券）枚举，但未实现。

**发放优先级设计**（预留，当前仅积分）：
1. 优先发放优惠券（`reward_type = 'coupon'`）
2. 积分补充（`reward_type = 'integral'`）

### 5.2 GrantRewardInner 发放流程

```php
private static function GrantRewardInner($reward_id)
```

1. 查询奖励记录（status=0 PENDING 待发放），加行锁
2. 如果 `reward_type = 'integral'` 且 `reward_value > 0`：
   - 查询邀请人信息
   - 增加邀请人积分：`Db::name('User')->inc('integral', reward_value)->update()`
   - 记录积分日志：调用 `IntegralService::UserIntegralLogAdd()`
3. 更新奖励记录状态：`status → 1`（GRANTED，已发放），`upd_time → 当前时间`

### 5.3 后台手动补发

```php
public static function AdminGrant($params)
```

- 仅允许对 `status=PENDING(0)/CANCELLED(2)/REVOKED(3)` 且 `reward_value > 0` 的记录补发
- 非 PENDING 状态先重置为 PENDING，再调用 `GrantReward()`

### 5.4 后台撤销

```php
public static function AdminCancel($params)
```

- 将 GRANTED 状态的记录改为 REVOKED(3)，并扣回已发放积分

### 5.3 积分日志

发放积分时通过 `IntegralService::IntegralLogAdd()` 写入 `sxo_integral_log` 表，确保积分变动可追溯。

## 六、sxo_invite_reward 表结构

| 列名 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| id | int unsigned AUTO_INCREMENT | - | 自增ID |
| inviter_id | int unsigned | 0 | 邀请人用户ID |
| invitee_id | int unsigned | 0 | 被邀请人用户ID |
| reward_type | char(30) | 'integral' | 奖励类型（integral积分/coupon优惠券） |
| reward_value | int unsigned | 0 | 奖励值（积分数/优惠券ID） |
| trigger_event | char(30) | 'register' | 触发事件（register注册/first_order首单） |
| status | tinyint unsigned | 0 | 状态（0待发放/1已发放/2已取消/3已撤销） |
| add_time | int unsigned | 0 | 添加时间戳 |
| upd_time | int unsigned | 0 | 更新时间戳 |

**索引**：PRIMARY(id), KEY(inviter_id), KEY(invitee_id), KEY(trigger_event), KEY(status)

**trigger_event 枚举值**：

| 值 | 说明 |
|----|------|
| register | 新用户注册 |
| first_order | 被邀请人首单完成 |

**reward_type 枚举值**：

| 值 | 说明 |
|----|------|
| integral | 积分奖励（已实现） |
| coupon | 优惠券奖励（预留，未实现） |

**status 枚举值**：

| 值 | 说明 |
|----|------|
| 0 | 待发放 (PENDING) |
| 1 | 已发放 (GRANTED) |
| 2 | 已取消 (CANCELLED) |
| 3 | 已撤销 (REVOKED) |

## 九、防刷/幂等设计说明

### 9.1 幂等设计

| 场景 | 幂等机制 | 实现 |
|------|----------|------|
| 注册奖励 | 唯一记录校验 | 查询 `inviter_id + invitee_id + trigger_event=register` 是否已存在 |
| 首单奖励 | 唯一记录校验 | 查询 `inviter_id + invitee_id + trigger_event=first_order` 是否已存在 |
| 积分发放 | 状态校验 | GrantReward 仅处理 status=0 的记录 |
| 签到 | 状态校验 | SignupCheckin 仅处理 checkin_status=0 的记录 |

**关键代码**：

```php
// 注册幂等
$exists = Db::name('InviteReward')->where([
    ['inviter_id', '=', $inviter_id],
    ['invitee_id', '=', $params['user_id']],
    ['trigger_event', '=', 'register'],
])->find();
if (!empty($exists)) {
    return DataReturn('已存在注册邀请记录', 0);
}
```

### 9.2 防刷设计

| 风险 | 当前防护 | 后续优化 |
|------|----------|----------|
| 自己邀请自己 | inviter_id == user_id 校验 | 已实现 |
| 虚假注册刷积分 | 无 | 需增加：注册后X天内不能提现积分、新用户需完成首单才发放注册奖励 |
| 邀请码暴力破解 | MD5哈希，不可逆 | 已足够安全 |
| 重复领取首单奖励 | 幂等校验 | 已实现 |
| 批量注册小号 | 无 | 需增加：手机号验证码注册、IP频率限制、设备指纹 |

### 9.3 当前MVP阶段的安全边界

- 邀请码使用MD5哈希，不可逆推，安全性足够
- 幂等校验覆盖了重复发放场景
- 缺少手机号验证、IP限制等防刷手段，MVP阶段可接受
- 奖励配置默认值为0（不发放），需手动配置后才生效，降低了误操作风险

### 9.4 建议的后续加固

1. **注册防刷**：接入短信验证码，限制同一IP注册频率
2. **奖励延迟发放**：注册奖励延迟24小时发放，首单奖励在订单完成后延迟发放
3. **异常检测**：同一邀请人短时间内大量新用户注册时触发告警
4. **邀请码优化**：新增 `sxo_user.invite_code` 字段，避免全表遍历反查
5. **优惠券奖励**：实现 `reward_type=coupon` 的发放逻辑
